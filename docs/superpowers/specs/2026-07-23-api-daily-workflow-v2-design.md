# API 日常测试工作流 V2 设计

## 背景

现有 API 闭环已经具备多 Apifox 来源、不可变接口版本、模块范围、异步 AI
生成、MeterSphere 3.6.5 执行绑定、业务鉴权引用和真实报告，但页面仍以底层
对象为中心：

- 资产页同时承担来源配置、同步状态、版本和接口选择，用户无法判断同步是否会自动发生。
- 计划页直接展示来源 ID、版本 ID、全部模块路径、全部接口和全部用例；977 个接口、
  1590 条用例会形成不可审阅的信息墙。
- 鉴权元数据绑定在单个 Apifox 来源上，即使多个来源使用同一个 MeterSphere 环境，
  用户仍会看到“尚未配置”并重复提交 Token。
- 资产、生成、确认、执行和报告是五个独立页面，缺少统一的当前范围、完成状态和
  下一步操作。

本轮目标不是重写 API 执行引擎，而是把已有能力组织成测试人员能够直接理解和操作的
日常工作流。

## 官方产品对照

本设计只采用官方文档中已经稳定使用的模式：

- Postman 以 Workspace 组织 API、Collection、Environment 和运行记录；Collection
  或 Folder 是执行范围，运行后按请求查看测试结果、错误和 Console Log。
  [Workspaces](https://learning.postman.com/docs/collaborating-in-postman/using-workspaces/overview)
  [Scheduled runs](https://learning.postman.com/docs/tests-and-scripts/running-collections/viewing-scheduled-collection-runs/)
- Apifox 使用项目、模块、环境变量和场景用例组织自动化；同一个 Token 应通过环境
  变量和全局 Header 复用，场景用例负责组合接口、变量传递和执行报告。
  [环境与变量](https://docs.apifox.com/global-environment-session-variables)
  [自动化测试](https://docs.apifox.com/automated-testing)
- MeterSphere 把定义、用例、场景和报告分开，并在环境级提供项目通用请求头、环境
  变量和场景前置处理。
  [接口测试概述](https://metersphere.io/docs/v3.x/user_manual/api_test/intro/)
  [环境管理](https://metersphere.io/docs/v3.x/user_manual/project_management/environment/)

因此平台采用“项目/业务线确定范围、环境提供公共鉴权、小批量生成、按接口审阅、
按运行查看报告”的组合，而不是把三个产品的配置表单直接拼在一起。

## 设计原则

1. 默认页面只回答三个问题：现在测试什么、还缺什么、下一步做什么。
2. `source_id`、`revision_id`、`auth_ref`、`binding_fingerprint` 和 AI trace ID
   只出现在可展开的技术详情中。
3. 每个页面只有一个主操作；刷新、设置、重试和清除属于辅助操作。
4. 任何自动行为都展示最近结果和下次动作，不使用“已开启”但不可验证的模糊状态。
5. 过期、待补数据和未鉴权必须同时展示原因和可执行的修复入口。
6. 大批量数据先摘要、再筛选、最后按需展开；不默认渲染全量明细。
7. 密钥只写入 MeterSphere 环境变量，平台、浏览器、日志、计划和报告均不保存明文。

## 信息架构

所有 API 页面顶部共享一个五步流程条：

```text
1 选择业务  ->  2 同步接口  ->  3 生成用例  ->  4 审阅确认  ->  5 执行报告
```

流程条固定显示：

- 当前 Apifox 项目名称。
- 当前业务线名称。
- 当前活动版本的人类可读时间。
- 当前步骤状态：未开始、进行中、待处理、已完成、已过期。
- 唯一的下一步主操作。

流程条不是向导锁。用户可以返回前一步，但切换项目、业务线或版本后，后续状态必须按
服务端事实重新计算。

## 项目、业务线和模块

### 项目

现有 `source_id` 继续作为 Apifox 项目工作区边界。UI 只显示来源名称和 Apifox 项目
名称；内部 ID 放入技术详情。

### 业务线

业务线不新增人工维护字段，而是从活动 revision 的模块路径第一段确定性派生：

```text
家用业务/app接口/我的/我的收藏 -> 家用业务
共享业务/商户小程序/订单       -> 共享业务
地铁业务/模型库中台/模型分类   -> 地铁业务
```

`未分组` 作为明确的兜底业务线。业务线只用于筛选和展示，不替代 `source_id`、
`endpoint_key` 或 revision 所有权。

### 模块

选定业务线后只展示该业务线下的模块树。选中模块表示缩小浏览范围，不等于选择该模块
全部接口。接口勾选始终是显式动作。

单次 AI generation 仍限制 1-60 个接口。当前筛选结果超过 60 时：

- “选择当前结果”不可用。
- 页面提示当前数量和 60 条上限。
- 用户必须继续选择子模块、搜索接口或按 method 过滤。
- 平台不擅自截取前 60 条。

## 接口资产与自动同步

### 默认状态

已配置来源默认开启定时同步。资产页的首要状态改为：

```text
接口已自动更新
最近成功：2026-07-23 22:05
下次检查：23:05
本次变化：新增 3 / 修改 2 / 删除 0
```

来源新建成功或同步相关配置发生变化时，服务端立即排队一次同步。普通页面刷新不重复
触发同步；后续由现有 scheduler 按 `sync_interval_minutes` 执行。

“立即同步”是辅助操作，用于用户需要马上获取最新定义时。同步失败必须保留上一活动
revision，并展示失败原因和“重试同步”操作。

### 配置

来源配置移动到侧边抽屉，默认页面不展开 Token、Project ID、Branch ID 等字段。
配置保存成功后关闭抽屉，主页面显示安全摘要：

- 项目名称。
- 凭据已保存。
- 自动同步周期。
- 同步范围：全部模块或 N 个模块。

## 环境级公共鉴权

### 身份

公共鉴权 Profile 的身份由以下非秘密字段确定：

```text
MeterSphere connection identity + project_id + environment_id
```

同一 MeterSphere 项目和环境下的所有 Apifox 来源、业务线和模块共享一个 Profile。
本地 Profile 只保存：

- `auth_ref`
- `auth_type`
- `header_name`
- `variable_name`
- `project_id`
- `environment_id`
- `configured`
- 时间和非可逆 fingerprint

明文仍只写入 MeterSphere 环境变量。

### 自动复用

来源绑定 MeterSphere 项目和环境后，执行上下文自动查找对应 Profile：

- 已存在：直接返回“已复用测试环境鉴权”，不显示 Token 输入框。
- 不存在：显示一次“配置测试环境鉴权”。
- 切换到另一个环境：重新按新环境查找，不沿用旧环境密钥。

现有来源级 `auth_binding` 文件作为兼容输入。首次读取时若环境级 Profile 不存在，
可把其非秘密元数据迁移为 Profile；不读取、不复制、不返回远端秘密。

### 更新与清除

“更新鉴权”替换同一环境变量的值，Profile 身份保持不变。清除是影响该环境所有来源的
重要操作，UI 必须明确显示影响范围并二次确认。清除远端变量成功后才删除本地 Profile。

## AI 用例生成

### 生成前

计划页不再显示全部接口表。顶部展示生成范围摘要：

```text
家用业务 / 我的收藏
已选择 8 个接口
预计 1 个 AI 批次
将生成：正向用例、必填字段校验、文档已声明的鉴权校验
```

主操作命名为“生成 AI 用例”。点击后服务端仍按最多 12 个接口顺序分批。

### 生成中

默认进度采用四个用户阶段：

```text
校验接口范围 -> AI 分批设计 -> 平台可执行性校验 -> 草稿已生成
```

批次、模型、trace、provider、原始错误和 generation ID 放入“技术详情”。技术详情的
展开状态和滚动位置在轮询时保持不变。

### 生成结果

生成完成后立即打开最新草稿，不要求用户再从历史计划列表中寻找。部分失败时保留成功
批次，并提供“重试失败批次”；不能把规则种子伪装成 AI 成功。

## 用例审阅

### 摘要

计划详情先展示：

- 接口数。
- 用例数。
- 可执行数。
- 待补数据数。
- 版本是否最新。
- 当前环境鉴权是否就绪。

缺失字段按类型聚合，例如“请求数据 18 条、断言 4 条、鉴权 1 条”，不把几十个字段
路径一次性铺开。点击聚合项后过滤受影响用例。

### 按接口分组

用例按 endpoint 分组为折叠行：

```text
POST /print3d/api/v1/sign/testPost  测试接口签名
3 条用例 · 1 条可执行 · 2 条待补数据
```

展开后每条用例显示：

- 用例名称和类型中文标签。
- 请求 method/path。
- 已准备的数据项数量。
- 鉴权来源。
- 断言摘要。
- 待补字段和原因。

原五列表格不再作为默认视图。支持“全部、可执行、待补数据、已变化”筛选和名称/路径
搜索。大量分组只渲染当前分页，默认每页 20 个 endpoint。

### 操作

主操作根据状态唯一变化：

- 草稿且可确认：`确认计划`
- 草稿且待补数据：`查看待补数据`
- 已确认且最新：`去执行`
- 已过期：`按最新接口重新生成`

“重新生成全部”和技术详情作为辅助操作。

## MeterSphere 执行与报告

执行页顶部继承项目、业务线和环境。公共鉴权只显示紧凑状态：

```text
APP 测试环境 · Bearer 鉴权已就绪 · 该环境公共配置
```

仅在没有 Profile 时展示“配置鉴权”。已确认且最新的计划显示主操作“推送并执行”。
运行开始后继续使用现有四阶段真实状态和真实报告轮询，不增加前端伪状态。

报告页按当前来源和业务线筛选，默认显示本次运行摘要；请求级响应、错误和技术日志按需
展开并保留滚动位置。

## 状态与下一步规则

服务端事实优先级：

1. 未配置来源：`选择业务`
2. 无活动 revision：`同步接口`
3. 未选择接口：`选择模块和接口`
4. generation 运行中：`查看生成进度`
5. 有草稿待确认：`审阅用例`
6. 草稿缺数据：`补充测试数据`
7. 已确认但过期：`重新生成`
8. 已确认且可执行：`推送并执行`
9. execution 运行中：`查看执行进度`
10. execution 终态：`查看报告`

前端不得根据按钮点击自行宣称步骤完成。

## 响应式和可访问性

- 桌面端流程条横向显示；390px 移动端改为当前步骤摘要和可展开完整步骤。
- 不允许标题、method、path、用例名称按单字换行。
- 图标按钮使用现有项目图标能力并提供 `aria-label`/tooltip。
- 用例详情、技术日志和长 JSON 使用独立滚动容器，不推动整个页面横向溢出。
- 轮询不得重置展开项、筛选、分页或滚动位置。

## 兼容和非目标

- 保留现有 source、revision、plan、generation、execution 和 report ID。
- 保留现有 authenticated routes；来源级 auth route 改为环境 Profile 的兼容入口。
- 不修改 UI Agent、Midscene YAML、Runner、Sonic、scorer 或历史任务 YAML。
- 不在本轮实现 Phase D/E/F、RBAC、Outbox、通知或统一资产索引。
- 不改变千问模型选择策略。

## 验收标准

1. 新建/修改 Apifox 来源后自动排队同步，并显示最近成功和下次检查。
2. 971 个接口能按业务线和模块浏览；计划页默认不渲染 971 行。
3. 单次最多选择 60 个接口，超过时不擅自截断并给出明确操作提示。
4. 同一 MeterSphere 项目/环境的第二个来源自动复用鉴权，不再次要求 Token。
5. 默认 UI 不出现内部 source/revision/auth/trace ID。
6. 计划按 endpoint 折叠审阅，长名称和路径不出现单字竖排。
7. 生成过程、计划状态和下一步操作均来自真实服务端状态。
8. 桌面端和 390px 移动端无页面级横向溢出。
9. 现有跨来源隔离、CAS、重启恢复、密钥不落盘和报告归属测试继续通过。
